## Purpose

Define a minimal protocol for discovering, downloading, and verifying signed skills and yapCAD packages from any git-hosted repository. The protocol is transport-agnostic — it works with Gitea, GitHub, GitLab, or any git hosting with a web API. Future backends (IPFS, custom) can implement the same interface.

## Design Principles

1. **Git-native.** The repo IS a git repository. No custom server required. The protocol layers conventions on top of standard git hosting APIs.
2. **Signature-first.** Every artifact must be signed. Unsigned artifacts are invisible to the client.
3. **Multi-repo.** Clients maintain a list of repo sources. Trust is per-signer, not per-repo.
4. **Minimal API surface.** A repo needs to support exactly 5 operations. If your git host has a REST API, you can be a skill repo.
5. **Offline-capable.** A git clone of the repo contains everything needed — no external dependencies for verification.

## Repository Layout

A skill repo is a git repository with this structure:

```
skill-repo/
├── registry.json              # Package index (auto-generated)
├── keys/                      # Signer public keys
│   ├── <signer-id>.pub        # Ed25519 public key
│   └── ...
├── packages/
│   ├── <package-name>/
│   │   ├── MANIFEST.json      # Package metadata
│   │   ├── MANIFEST.sig.json  # Detached signature
│   │   ├── <artifact>.tar.gz  # Signed artifact (skill or .ycpkg)
│   │   └── README.md          # Human-readable description
│   └── ...
└── .github/                   # or .gitea/
    └── workflows/
        └── verify.yml         # CI: verify signatures on push
```

### `registry.json`

The package index. Auto-generated by CI on every push. Clients fetch this first.

```json
{
  "version": 1,
  "repo": "dml-internal",
  "updated": "2026-02-19T18:00:00Z",
  "packages": [
    {
      "name": "yapcad-printing",
      "version": "1.0.0",
      "type": "skill",
      "description": "3D printing pipeline for yapCAD",
      "signer": "garrett@dml",
      "signed_at": "2026-02-19T17:30:00Z",
      "tags": ["printing", "3d", "bambu", "prusa"],
      "license": "MIT",
      "path": "packages/yapcad-printing",
      "artifact": "packages/yapcad-printing/yapcad-printing-1.0.0.tar.gz",
      "manifest": "packages/yapcad-printing/MANIFEST.json",
      "signature": "packages/yapcad-printing/MANIFEST.sig.json",
      "sha256": "a1b2c3d4..."
    },
    {
      "name": "bicycle-frame",
      "version": "0.3.0",
      "type": "ycpkg",
      "description": "Parametric bicycle frame with mini-skill for rider fitting",
      "signer": "rich@dml",
      "signed_at": "2026-02-19T16:00:00Z",
      "tags": ["bicycle", "frame", "parametric", "mini-skill"],
      "license": "Apache-2.0",
      "path": "packages/bicycle-frame",
      "artifact": "packages/bicycle-frame/bicycle-frame-0.3.0.ycpkg",
      "manifest": "packages/bicycle-frame/MANIFEST.json",
      "signature": "packages/bicycle-frame/MANIFEST.sig.json",

```

### `MANIFEST.json`

Per-package metadata. This is what gets signed.

```json
{
  "name": "yapcad-printing",
  "version": "1.0.0",
  "type": "skill",
  "description": "3D printing pipeline for yapCAD",
  "author": "Garrett",
  "license": "MIT",
  "tags": ["printing", "3d", "bambu", "prusa"],
  "requires": {
    "yapcad": ">=2.0.0",
    "openclaw": ">=2026.2"
  },
  "depends": [
    {
      "name": "cad3d",
      "version": ">=1.0.0 <2.0.0",
      "signer": "garrett@dml",
      "key_hash": "sha256:a1b2c3..."
    }
  ],
  "artifact": "yapcad-printing-1.0.0.tar.gz",
  "artifact_sha256": "a1b2c3d4...",
  "external_artifacts": [],
  "files": {
    "SKILL.md": "sha256:...",
    "scripts/slice.py": "sha256:...",
    "scripts/print_bambu.py": "sha256:..."
  },
  "mini_skill": null,
  "created": "2026-02-19T17:30:00Z"
}
```

For `.ycpkg` packages with mini-skills, the `mini_skill` field contains the embedded skill definition (or a reference to the file within the package).

### `MANIFEST.sig.json`

Detached signature over the MANIFEST.json content. Compatible with the existing `skill-signer` format.

```json
{
  "signed_by": "garrett@dml",
  "algorithm": "ed25519",
  "public_key": "ssh-ed25519 AAAA...",
  "signature": "base64-encoded-signature",
  "signed_at": "2026-02-19T17:30:00Z",
  "manifest_sha256": "sha256-of-manifest-json"
}
```

## Client Configuration

Clients maintain a list of repos in their config:

```yaml
# ~/.yapcad/repos.yaml  (or OpenClaw skill config)
repos:
  - name: dml-internal
    url: https://172.23.23.141:3000/dml/skill-repo
    type: gitea
    priority: 10
    trust: [garrett@dml, rich@dml, jarvis@dml, jeremy@dml]

  - name: dml-public
    url: https://github.com/darkmatterlab/skills
    type: github
    priority: 20
    trust: [garrett@dml, rich@dml]

  - name: community
    url: https://github.com/openclaw-skills/registry
    type: github
    priority: 50
    trust: []  # empty = verify signature but don't auto-install
```

### Trust Model

- **Trusted signers** (`trust` list): Packages from these signers can be auto-installed.
- **Known signers** (key in `keys/`): Packages are verified but require user approval.
- **Unknown signers**: Packages are visible in search but flagged as unverified.
- **Unsigned packages**: Invisible. The client never shows them.

Trust is per-signer, not per-repo. If you trust `garrett@dml`, you trust his packages regardless of which repo they're in. This makes trust portable across the multi-repo architecture.

## Protocol Operations

### 1. `list` — Discover packages

Fetch `registry.json` from the repo's default branch.

**Git API mapping:**
- GitHub: `GET /repos/{owner}/{repo}/contents/registry.json`
- Gitea: `GET /api/v1/repos/{owner}/{repo}/contents/registry.json`
- GitLab: `GET /api/v4/projects/{id}/repository/files/registry.json/raw`
- Fallback: `git archive --remote=<url> HEAD registry.json`

**Client behavior:**
1. Fetch `registry.json` from each configured repo
2. Merge results, deduplicate by `(name, version)`, prefer higher-priority repo
3. Filter by trust level (hide unsigned, flag unverified)
4. Cache locally with TTL (default: 1 hour)

### 2. `search` — Find packages by keyword

Client-side operation on the cached `registry.json`:
- Match against `name`, `description`, `tags`
- Filter by `type` (skill, ycpkg, or both)
- Sort by relevance, then priority

No server-side search needed — the registry is small enough to filter locally.

### 3. `fetch` — Download a package

Download the artifact and its signature.

**Git API mapping:**
- GitHub: `GET /repos/{owner}/{repo}/contents/{artifact_path}` (base64) or raw download via releases
- Gitea: `GET /api/v1/repos/{owner}/{repo}/raw/{branch}/{artifact_path}`
- GitLab: `GET /api/v4/projects/{id}/repository/files/{path}/raw`
- Fallback: `git archive --remote=<url> HEAD {path}`

**Client behavior:**
1. Download `MANIFEST.json`, `MANIFEST.sig.json`, and the artifact
2. Verify artifact SHA256 matches manifest
3. Verify signature (see operation 4)
4. If verified, extract/install; if not, reject

### 4. `verify` — Check signature

Local operation using the signer's public key.

**Steps:**
1. Look up signer's public key: first check local keyring, then fetch from repo's `keys/{signer-id}.pub`
2. Verify `MANIFEST.sig.json` signature over `MANIFEST.json` content
3. Verify each file's SHA256 against `files` map in manifest
4. Verify artifact SHA256 against `artifact_sha256`
5. Return: `verified | untrusted_signer | invalid_signature | tampered`

**Key resolution order:**
1. Local trusted keyring (`~/.yapcad/trusted_keys/`)
2. Repo's `keys/` directory
3. GitHub public keys (`github.com/{user}.keys`) — Phase 1 PKI fallback

### 5. `publish` — Upload a package

Push a signed package to a repo. This is a git operation.

**Steps:**
1. Agent signs the package using `skill-signer` (or unified signing tool)
2. Agent creates/updates the package directory in the repo
3. Agent pushes via git (authenticated — SSH key or token)
4. CI (Git Actions) runs:
   - Signature verification
   - Security scan (no embedded secrets, no dangerous operations)
   - Registry regeneration (`registry.json` updated)
5. If CI passes, package is live

**For repos the user doesn't have push access to:**
- Fork → push to fork → submit pull request
- Maintainers review and merge
- Same as any open-source contribution workflow

## Git Actions — CI Pipeline

### `verify.yml` — Runs on every push to `packages/`

```yaml
name: Verify Skills
on:
  push:
    paths: ['packages/**']

jobs:
  verify:
    runs-on: ubuntu-latest  # or gitea-runner
    steps:
      - uses: actions/checkout@v4

      - name: Verify signatures
        run: |
          for manifest in packages/*/MANIFEST.json; do
            dir=$(dirname "$manifest")
            skill-signer verify "$dir" --keyring keys/
          done

      - name: Security scan
        run: |
          # Check for embedded secrets
          skill-signer audit "$dir" --check-secrets --check-dangerous

      - name: Regenerate registry
        run: |
          skill-signer registry generate --output registry.json

      - name: Commit registry
        run: |
          git add registry.json
          git commit -m "chore: regenerate registry" || true
          git push
```

## Versioning

Packages use semver (`MAJOR.MINOR.PATCH`). The registry can contain multiple versions of the same package. Clients default to latest, but can pin versions.

**Version policy:**
- Breaking changes → major bump
- New features → minor bump
- Bug fixes → patch bump
- The registry retains all versions (git history provides full lineage)

## Large Artifacts

In-repo artifacts are limited to **50MB**. For larger assets (trained models, large meshes, compiled binaries), use the `external_artifacts` field in `MANIFEST.json`:

```json
"external_artifacts": [
  {
    "name": "trained-model.onnx",
    "url": "https://releases.example.com/model-v1.0.0.onnx",
    "sha256": "e5f6a7b8...",
    "size": 128000000,
    "required": true
  }
]
```

**Client behavior:**
1. Download external artifacts after verifying the manifest signature
2. Verify SHA256 of each downloaded artifact against the manifest
3. If any required artifact fails verification, reject the entire package
4. Cache external artifacts locally (keyed by SHA256) to avoid re-downloads

This keeps the git repo lean while still providing hash-verified integrity for large files.

## Two-Tier PKI

The signing infrastructure uses a **cold root key** and **hot operational keys** to balance security with automation.

### Key Hierarchy

```
Root Key (COLD — offline, air-gapped)
├── Signs: signer public keys (onboarding)
├── Signs: CI registry key (delegation)
├── Signs: revocations (emergency)
│
├── Signer Keys (WARM — developer machines)
│   ├── garrett@dml — signs packages
│   ├── rich@dml — signs packages
│   ├── jarvis@dml — signs packages (agent)
│   └── jeremy@dml — signs packages
│
└── CI Registry Key (HOT — runs in CI)
    └── Signs: registry.json only
```

### Key Lifecycle

1. **Root key generation:** Ed25519 keypair, generated offline, private key stored on hardware token or air-gapped machine. Public key distributed in repo as `keys/_root.pub`.
2. **Signer onboarding:** Developer generates keypair, submits `.pub` to repo. Root key signs a certificate binding `signer-id` to public key. Certificate stored as `keys/<signer-id>.cert.json`.
3. **CI key delegation:** Root key signs a restricted certificate for the CI key — only valid for signing `registry.json`, not packages.
4. **Key rotation:** New key generated, root signs new cert, old key added to revocation list.

### Revocation

A `revocations.json` file in the repo root, **signed by the root key**:

```json
{
  "version": 1,
  "signed_by": "_root",
  "revocations": [
    {
      "key_hash": "sha256:abc123...",
      "signer": "compromised@dml",
      "revoked_at": "2026-03-15T00:00:00Z",
      "reason": "Key compromise"
    }
  ],
  "signature": "base64-encoded-root-signature"
}
```

**Client behavior:**
1. Fetch `revocations.json` on every registry refresh
2. Verify the root signature on the revocations file
3. Reject any package signed by a revoked key, even if previously trusted
4. Cache revocations with a short TTL (15 minutes) to limit exposure window

If the CI key is compromised: root key publishes a revocation, rotates the CI key, and re-signs the registry. Since the CI key can only sign `registry.json` (not packages), the blast radius is limited to registry tampering — package signatures remain valid.

## Dependency Resolution

Packages can declare dependencies on other signed packages using semver ranges with signer pinning.

### Dependency Specification

```json
"depends": [
  {
    "name": "cad3d",
    "version": ">=2.2.0 <3.0.0",
    "signer": "bob@mcbobster.com",
    "key_hash": "sha256:abc123..."
  }
]
```

- **`name`** — Package name in the registry
- **`version`** — Semver range (npm-style: `>=`, `<`, `^`, `~`)
- **`signer`** — Required signer identity. Prevents name-squatting attacks where a different repo publishes a same-named package.
- **`key_hash`** — SHA256 of the signer's public key. Belt-and-suspenders: even if the signer ID is spoofed, the key must match.

### Resolution Algorithm

1. Build dependency graph from root package
2. For each dependency, find the highest version in configured repos that satisfies the range AND matches the signer+key_hash
3. Detect cycles (error) and diamonds (resolve to highest compatible version)
4. Download and verify all dependencies before installing root package

### Depth Limit

Maximum dependency depth: **3 levels**. Skills should be composable but not deeply nested. If you need more than 3 levels, refactor.

### Conflict Resolution

If two packages depend on different versions of the same dependency:
- If ranges overlap → use highest version in the intersection
- If ranges are incompatible → error, require user to choose

## Security Considerations

1. **No code execution during install.** Skills are static files (SKILL.md, scripts, configs). No `postinstall` hooks.
2. **Signature covers all files.** The manifest includes SHA256 of every file in the package. Tampering with any file invalidates the signature.
3. **CI is defense-in-depth.** Even if a signer is trusted, CI scans for secrets and dangerous patterns before the package enters the registry.
4. **Key revocation:** Remove the signer's `.pub` from `keys/` and push. All their packages become unverified. (Future: explicit revocation list.)
5. **Repo compromise:** Since trust is per-signer (not per-repo), compromising a repo doesn't help unless you also have a trusted signer's private key.

## Rate Limiting

### Client-side
- Default: **60 requests/minute** per repo
- Registry refresh: **1 hour** cache TTL (configurable)
- Revocation refresh: **15 minute** cache TTL (not configurable below this)
- Exponential backoff on 429 responses

### Server-side (Gitea)
- Configure in `app.ini`: `[api]` section, `MAX_RESPONSE_ITEMS` and rate limiting middleware
- Recommended: 100 req/min per IP for anonymous, 300 req/min for authenticated
- GitHub: handled automatically by their API rate limits (60/hr anonymous, 5000/hr authenticated)

## ClawHub Integration

The protocol extends the existing `clawhub` CLI rather than replacing it. ClawHub becomes a client implementation that speaks the skill repo protocol, with backward compatibility for existing unsigned skills.

### New Commands

```bash
# Repo management
clawhub repo add <name> <url> [--type gitea|github|gitlab] [--trust <signer-id>...]
clawhub repo remove <name>
clawhub repo list

# Search across all configured repos (signed + unsigned)
clawhub search <query> [--signed-only] [--type skill|ycpkg]

# Install with signature awareness
clawhub install <package>                    # Requires signature from trusted signer
clawhub install <package> --allow-unsigned   # ⚠️ WARNING: installing unsigned skill
clawhub install <package>@2.1.0             # Pin version

# Verification
clawhub verify <package>                     # Check signatures of installed package
clawhub verify --all                         # Verify all installed packages

# Publishing (for skill authors)
clawhub publish <dir> --sign-as <signer-id>  # Sign and push to configured repo
```

### Backward Compatibility

- Existing ClawHub registry treated as an **unsigned repo source** at lowest priority
- `clawhub install` from ClawHub registry requires `--allow-unsigned` flag
- Warning displayed: `⚠️ This skill is unsigned and unverified. Install at your own risk.`
- Skills installed from signed repos get a `✅ Verified` badge in `clawhub list`
- Migration path: ClawHub skill authors can sign their existing skills and publish to a signed repo

### Trust Precedence

When the same package exists in multiple repos:
1. Signed + trusted signer → auto-install
2. Signed + known signer → prompt for approval
3. Signed + unknown signer → show in search, flag as unverified
4. Unsigned (ClawHub legacy) → only with `--allow-unsigned`

## Resolved Design Decisions

| # | Question | Decision | Rationale |
|---|----------|----------|-----------|
| 1 | Large artifacts | 50MB in-repo limit + `external_artifacts` with SHA256 | Keeps git lean; hash verification for integrity |
| 2 | Registry signing | Yes, by hot CI key with two-tier PKI | Prevents transport-layer tampering; revocation via cold root key limits blast radius |
| 3 | Dependencies | Semver ranges + signer/key_hash pinning, 3-level depth limit | Enables skill composition without copying; signer pinning prevents name-squatting |
| 4 | Rate limiting | Client 60 req/min, Gitea 100-300 req/min | Reasonable defaults; exponential backoff on 429 |
| 5 | ClawHub alignment | Extend `clawhub` CLI with repo/verify/signing commands | Wider adoption; backward compat with `--allow-unsigned`; avoid ecosystem fragmentation |
---

*This protocol is designed to be the simplest thing that works, layered on infrastructure that already exists (git). Complexity can be added later — IPFS backends, on-chain PKI, payment rails — without changing the client-facing protocol.*
